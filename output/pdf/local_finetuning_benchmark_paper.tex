\title{A Comparative Local Fine-Tuning Benchmark Across Candle, Burn, PyTorch, and MLX on Apple Silicon}
\author{Hamze Ghalebi}
\date{April 2026}

\begin{abstract}
This paper describes a reproducible local benchmark for supervised fine-tuning across four training backends: Candle, Burn, PyTorch, and MLX. The benchmark is designed to answer a narrow engineering question rather than a broad research question: given a fixed chat dataset, a fixed held-out evaluation slice, and a fixed small-model training recipe, how much does fine-tuning improve behavior, and how do the training frameworks differ in quality, training wall time, and inference latency? Every backend saves a real base checkpoint, trains a real fine-tuned checkpoint, evaluates both on the same held-out examples, and emits the same report schema. On the current Apple Silicon setup, all four backends improve held-out ROUGE-L after fine-tuning. MLX achieves the largest ROUGE-L gain, the fastest training wall time, and the lowest fine-tuned inference latency on this benchmark.
\end{abstract}

\section{Introduction}

Local fine-tuning workflows are often evaluated through incomplete proxies: training loss alone, a single backend, or an external inference service standing in for the actual model under study. That makes it difficult to answer the most operationally relevant question: whether the model trained in the repository is measurably better after fine-tuning than before fine-tuning, and whether the answer changes as the backend changes.

Author: Hamze Ghalebi. This work is associated with the ``Hackathon: Benchmarking Small Language Models in the Real World,'' organized by AI Paris Thinker, but it does not use the official \texttt{polarsbench.net} evaluation platform, the hidden runner-provided evaluation dataset, or the official leaderboard scoring path. The benchmark described here is therefore an independent local benchmark rather than an official hackathon submission result.

This repository now implements a true pre-vs-post comparison for four backends:

\begin{itemize}
    \item Candle
    \item Burn
    \item PyTorch
    \item MLX
\end{itemize}

The contribution is practical rather than algorithmic. The system establishes one benchmark contract, one held-out evaluation path, and one report format that every backend must satisfy. That makes the comparison machine-checkable and reproducible.

\section{System overview}

The pipeline has two layers: a shared data and evaluation contract, and backend-specific training implementations. The separation is deliberate. It ensures that changing the framework does not change the task.

\begin{center}
\fbox{%
\parbox{0.94\linewidth}{%
\textbf{Pipeline:} raw chat JSONL $\rightarrow$ Rust data cleaning $\rightarrow$ train/valid JSONL $\rightarrow$ backend training path $\rightarrow$ base checkpoint + fine-tuned checkpoint $\rightarrow$ held-out evaluation $\rightarrow$ per-backend JSON report $\rightarrow$ consolidated leaderboard and SVG visualization
}%
}
\end{center}

The cleaned dataset preserves a conversational structure with a \texttt{messages} array. Evaluation examples are created by taking every message except the final assistant message as the prompt, and using the final assistant message as the target. This gives all backends the same generation task: produce a better assistant continuation from the same conversation prefix.

\section{Backend implementations}

\subsection{Candle}

The Candle path is implemented entirely in Rust and stores checkpoints in \texttt{safetensors} format. It serves as the Rust-native baseline that proves the benchmark can run without Python.

\subsection{Burn}

The Burn path is also implemented entirely in Rust and stores checkpoints in \texttt{mpk} format. For this benchmark, decoder-only behavior is modeled with an encoder stack plus a causal attention mask. That choice preserves causal training behavior while fitting Burn's current transformer surface more cleanly for this local benchmark.

\subsection{PyTorch}

The PyTorch path runs in a dedicated Python 3.12 environment managed by \texttt{uv}. It uses token embeddings, positional embeddings, pre-norm multi-head self-attention blocks, causal masking, and AdamW, with MPS acceleration on Apple Silicon when available.

\subsection{MLX}

The MLX path also runs in the pinned Python 3.12 environment. It uses \texttt{mlx.nn.Embedding}, \texttt{mlx.nn.MultiHeadAttention}, additive causal masks, \texttt{nn.value\_and\_grad}, and AdamW. It is the backend most directly aligned with Apple Silicon hardware in this repository.

\section{Experimental setup}

The benchmark fixes the following conditions across backends:

\begin{itemize}
    \item Training split: \texttt{data/train\_mistral.jsonl}
    \item Validation split: \texttt{data/valid\_mistral.jsonl}
    \item Training rows: 512
    \item Held-out evaluation rows: 20
    \item Training steps: 800
    \item Maximum sequence length: 128
    \item Model dimension: 128
    \item Attention heads: 4
    \item Layers: 3
    \item Hidden dimension: 512
    \item Maximum generated tokens: 48
\end{itemize}

All four backends use the same intentionally simple word-level tokenizer family. This is a tradeoff. It does not aim to approximate production tokenization. It exists to keep the benchmark self-contained and comparable without depending on external model assets or backend-specific tokenizer stacks.

\section{Evaluation protocol}

Every backend follows the same sequence:

\begin{enumerate}
    \item Initialize a base model.
    \item Save a real base checkpoint.
    \item Fine-tune that same model on the training split.
    \item Save a real fine-tuned checkpoint.
    \item Reload both checkpoints.
    \item Evaluate both on the same held-out examples.
    \item Emit a report with the same JSON schema.
\end{enumerate}

The shared metrics are:

\begin{itemize}
    \item exact match
    \item ROUGE-L
    \item response length
    \item latency
\end{itemize}

The exact match metric remains useful as a hardness indicator, but ROUGE-L is the primary quality measure because the target task is open-ended assistant continuation rather than strict string reproduction.

The main observations are straightforward:

\begin{enumerate}
    \item All four backends improve held-out ROUGE-L after fine-tuning.
    \item MLX achieves the highest ROUGE-L delta.
    \item MLX trains fastest on this machine.
    \item MLX also yields the lowest fine-tuned inference latency.
    \item PyTorch is a strong second option, outperforming both Rust-native backends on ROUGE-L delta and fine-tuned latency in this setup.
\end{enumerate}

Exact match remains at 0.0 across all runs. That is expected for a small local causal language model on a limited held-out slice with open-ended continuation targets. In this regime, exact match is too brittle to serve as the primary metric; ROUGE-L is the more informative signal.

\section{Discussion}

The most important result is not the ranking itself. The most important result is that the pipeline can now answer the before-vs-after question with real checkpoints and held-out evaluation across multiple frameworks. That turns the repository from a data-preparation prototype into an empirical fine-tuning benchmark.

Three engineering implications follow.

First, the benchmark is sensitive enough to detect real post-training improvement. All four backends move in the right direction on ROUGE-L, which indicates that the evaluation setup is not random noise.

Second, the backend ranking is multi-dimensional. Training speed alone is not sufficient. The system measures quality delta, training wall time, response shape, and inference latency, which is closer to the tradeoff surface engineers actually care about.

Third, the current machine matters. MLX is especially well matched to Apple Silicon, and the results support that expectation rather than merely assuming it. The benchmark therefore acts as a local systems report as much as a model-quality report.

\section{Threats to validity}

The current benchmark has real limits.

\begin{itemize}
    \item The model is intentionally small. These results do not estimate the behavior of a large production-scale checkpoint.
    \item The tokenizer is simplified. That keeps the benchmark reproducible, but it lowers the absolute quality ceiling.
    \item The held-out evaluation slice is small at 20 rows. The ranking is therefore directional rather than final.
    \item The implementations are behaviorally aligned, not numerically identical. They share the same benchmark contract, but not the same low-level execution graph.
\end{itemize}

For that reason, the correct interpretation is narrow: the benchmark shows how well each backend performs on this local causal-language-model setup, on this machine, under this fixed recipe.

\section{Reproducibility}

The paper is based on repository artifacts generated from the following commands:

\begin{verbatim}
cargo run --features candle --bin 07_candle_true_pre_post -- \
  --train-path data/train_mistral.jsonl \
  --valid-path data/valid_mistral.jsonl \
  --run-dir artifacts/candle-true-pre-post \
  --train-limit 512 --eval-limit 20 --steps 800 \
  --max-seq-len 128 --d-model 128 --n-layers 3 \
  --mlp-hidden 512 --max-new-tokens 48

cargo run --features burn --bin 08_burn_true_pre_post -- \
  --train-path data/train_mistral.jsonl \
  --valid-path data/valid_mistral.jsonl \
  --run-dir artifacts/burn-true-pre-post \
  --train-limit 512 --eval-limit 20 --steps 800 \
  --max-seq-len 128 --d-model 128 --n-heads 4 \
  --n-layers 3 --mlp-hidden 512 --max-new-tokens 48

.venv-ml/bin/python scripts/true_pre_post_pytorch.py \
  --train-path data/train_mistral.jsonl \
  --valid-path data/valid_mistral.jsonl \
  --run-dir artifacts/pytorch-true-pre-post \
  --train-limit 512 --eval-limit 20 --steps 800 \
  --max-seq-len 128 --d-model 128 --n-heads 4 \
  --n-layers 3 --mlp-hidden 512 --max-new-tokens 48

.venv-ml/bin/python scripts/true_pre_post_mlx.py \
  --train-path data/train_mistral.jsonl \
  --valid-path data/valid_mistral.jsonl \
  --run-dir artifacts/mlx-true-pre-post \
  --train-limit 512 --eval-limit 20 --steps 800 \
  --max-seq-len 128 --d-model 128 --n-heads 4 \
  --n-layers 3 --mlp-hidden 512 --max-new-tokens 48

cargo run --features "candle burn" --bin 10_compare_all_backends -- \
  --report artifacts/candle-true-pre-post/true_pre_post_report.json \
  --report artifacts/burn-true-pre-post/true_pre_post_report.json \
  --report artifacts/pytorch-true-pre-post/true_pre_post_report.json \
  --report artifacts/mlx-true-pre-post/true_pre_post_report.json \
  --out-dir artifacts/all-backend-comparison
\end{verbatim}

\section{Conclusion}

This repository now supports a true comparative fine-tuning benchmark across Candle, Burn, PyTorch, and MLX. The benchmark produces real pre-training and post-training checkpoints, evaluates them on the same held-out examples, and summarizes the results in a backend-agnostic report schema.

Within this setup, MLX is currently the strongest backend on Apple Silicon. It yields the highest held-out ROUGE-L delta, the fastest training wall time, and the lowest fine-tuned inference latency. PyTorch is the strongest alternative, while Candle and Burn remain valuable Rust-native baselines for local experimentation and pipeline development.

The more general contribution is the benchmark method itself. By standardizing the dataset split, prompt construction, metrics, and report format, the repository now provides an evidence-producing framework for comparing local fine-tuning paths rather than a collection of disconnected training scripts.
